1. 项目基础理解 10 个问题
01 基础 为什么你的系统叫 Agent，而不是 Chatbot？ ▼
导师意图
考察对 Agent 核心概念的理解——是否清楚 Agent 和 Chatbot 在架构层面的本质区别，而非仅停留在名词层面。
推荐回答
Chatbot 收到问题后直接从模型训练数据生成回答——信息来源是模型记忆，无法验证。我的 Agent 不同：收到用户请求后，它不直接回答，而是进入 ReAct 循环——思考需要什么信息、通过 ToolRegistry 调用对应的 Tool 查询真实数据、观察结果后再决定下一步。系统有 6 个 Tool，每个 Tool 返回的数据都有明确来源。另外 Agent 的每一步决策都记录在 Trace 中——13 步可追溯。Chatbot 的输出没有数据来源，Agent 的输出可以追溯到具体的 Tool 调用。
可能追问

"Agent 的'智能'体现在哪里？是 LLM 智能还是系统智能？"

两个层面。LLM 层面——DeepSeek 理解自然语言，自主决定调用哪个 Tool、传什么参数；系统层面——EXECUTION_PLAN 定义了工作流逻辑，blocker_check 定义了安全约束，ToolRegistry 实现了 O(1) 动态调度。LLM 提供灵活性，系统提供可靠性。

02基础 你的系统和直接调用 DeepSeek API 有什么区别？▼
导师意图
考察是否理解"框架"与"API 调用"的区别——是否只是在 API 外包了一层皮。
推荐回答
直接调 DeepSeek API 只是把用户问题转发给大模型，模型从训练数据生成回答——没有数据验证。我的系统在 DeepSeek 之上构建了完整的 Agent 框架：LLMProvider 抽象层隔离 SDK 依赖，ToolRegistry 注册中心管理工具调度，ReAct 状态机控制推理循环，blocker_check 实现业务阻断，Trace 记录每一步决策。直接调 API 是一个函数调用，我的系统是一个有架构、有测试、有审计追踪的框架。
可能追问

"如果 DeepSeek API 挂了，你的系统还能用吗？"

能。系统同时实现了 RuleBasedReActAgent——基于规则化状态机的 Agent，完全离线运行，不依赖任何 LLM API。python demo.py 默认就是离线模式。LLM 模式和规则模式共享相同的 run() 接口和 ReActResult 返回类型。

2. 系统架构设计 15 个问题
11架构 请介绍你的整体系统架构。▼
导师意图
考察架构全局观——是否能清晰、完整地描述系统拓扑。
推荐回答
系统采用四层分离架构，从外到内依次是：Provider 层封装 LLM API 调用，当前实现 DeepSeek（OpenAI 兼容协议），通过 LLMProvider ABC 抽象使 Agent 不依赖具体 SDK。Agent 层包含 AgentRunner（总入口）、LLMReActAgent（ReAct 推理引擎）和 ToolRegistry（O(1) 工具调度中心）。Tools 层实现 6 个工具——4 个课程规划工具加 2 个 Demo 工具，以及 Scheduler 调度器和 ResourceProvider 资源提供器。Models 层定义 10 个 Pydantic 数据模型，字段名与 courses.json 全链路一致。依赖方向严格向下，零循环导入。
可能追问

"AgentRunner 和 LLMReActAgent 的职责边界在哪里？"

AgentRunner 是胶水层——它不包含业务逻辑。职责是：组装依赖（Registry/PromptLoader/Agent）、加载 System Prompt、调用 Agent.run()、格式化输出。LLMReActAgent 是引擎——负责 ReAct 循环的具体执行。

12架构 为什么采用分层架构？不分层会怎样？▼
导师意图
考察对分层设计价值的理解——是否知道每层存在的理由。
推荐回答
分层是为了隔离变化。举例：如果 Provider 和 Agent 耦合在一起，换一个 LLM（从 DeepSeek 换成 Claude）就要改 Agent 核心代码。分层后只需在 Provider 层新增一个 ClaudeProvider 类——Agent 层零改动。同样的道理，新增一个 Tool 只需在 Tools 层注册，Agent 层零改动。分层不是为了让架构图好看——每个层的边界都是一条"改动隔离线"。
13架构 AgentRunner 具体负责什么？为什么需要它？▼
导师意图
考察是否理解了"编排器"和"引擎"的职责分离。
推荐回答
AgentRunner 是薄胶水层——它的唯一职责是组件编排，不包含任何业务逻辑。具体做了五件事：初始化 Registry/PromptLoader/Agent（支持依赖注入）、调用 _validate_input() 做参数验证、通过 PromptLoader 加载和注入 System Prompt、委托 Agent.run() 执行 ReAct 循环、调用 format_result() 格式化输出。它不调用 Tool、不操作 FSM 状态、不做推理决策——这些是 LLMReActAgent 的职责。
14架构 为什么需要 LLMProvider 抽象？不能直接调 DeepSeek API 吗？▼
导师意图
考察对依赖倒置原则的理解——是否明白"面向接口编程"的工程价值。
推荐回答
可以直接调，但那样 Agent 代码就绑死在 DeepSeek 上了。LLMProvider 用 ABC 定义 chat(messages, tools) → LLMResponse 统一接口，Agent 只依赖这个接口。结果是：Agent 代码从头到尾没有 import openai，没有 import anthropic。换 Claude 只需实现 ClaudeProvider(LLMProvider)——Agent 一行不改。这就是依赖倒置：高层模块不依赖低层模块，两者都依赖抽象。
15架构 为什么 Tool 不能直接写在 Agent 里面？▼
导师意图
考察对模块解耦价值的理解。
推荐回答
写在 Agent 里面会导致三个问题：每次加 Tool 要改 Agent 代码（违反开闭原则）、Tool 无法独立测试（和 Agent 循环耦合）、Tool 无法被复用（比如 get_course_info 被 generate_learning_plan 编排器复用）。通过 ToolRegistry 注册后，Tool 是独立模块——可以单独测试、单独替换、被多个调用方复用。Agent 只知道 registry.invoke(name, **kwargs)，不关心具体有哪些 Tool。
16架构 Tool Registry 解决什么工程问题？不用它会怎样？▼
导师意图
考察是否理解 Registry Pattern 对比 if/elif 链的工程优势。
推荐回答
ToolRegistry 用 Python dict 实现 O(1) 工具查找，替代传统的 if/elif 条件链。三个工程优势：扩展性——加新 Tool 只需 registry.register(ToolEntry(...))，Agent 代码零改动；元数据管理——每个 Tool 的 name/description/input_schema/output_schema 统一存储，list_tools() 直接导出为 LLM Function Calling Schema；测试友好——可以注入 Mock Registry 测试 Agent，不需要真实 Tool 实现。
17架构 新增一个 Tool 需要修改哪些模块？▼
导师意图
验证"零改动"的声明是否真实——是否真的做到了开闭原则。
推荐回答
两步。第一步：在 src/tools/ 下实现工具函数，遵循签名 def my_tool(**kwargs) → dict，返回 {"success": true, "data": {...}} 或 {"success": false, "error": {...}}。第二步：在创建 Registry 时注册——registry.register(ToolEntry(name="my_tool", description="...", input_schema={...}, output_schema={...}, function=my_tool))。Agent 层（AgentRunner / LLMReActAgent / react_loop）、Provider 层、Models 层——全部不需要修改。
18架构 你的系统如何保证模块解耦？▼
导师意图
考察解耦的具体实现手段——不是喊口号，而是有代码证据。
推荐回答
四个层面。依赖注入：AgentRunner 构造函数接受可选的 registry/agent/prompt_loader，测试时可以注入 Mock。接口隔离：LLMProvider 是 ABC，Agent 只依赖这个抽象，不依赖 DeepSeek SDK。注册模式：Tool 通过 ToolRegistry 注册，Agent 通过 invoke(name, **kwargs) 调用——Agent 不知道具体有哪些 Tool。统一返回协议：所有 Tool 返回 {"success": bool, "data"/"error": {...}}，Agent 通过检查 success 字段做决策，不依赖 Tool 内部实现。
19架构 你的 Orchestrator 在哪里？▼
导师意图
考察是否理解编排器和工作流引擎的概念。
推荐回答
有两个层面的编排。Agent 层面：LLMReActAgent 是主编排器——它通过 ReAct 循环编排 Thought→ToolCall→Observation 的流程。Workflow 层面：generate_learning_plan 是课程规划的编排器——它编排 6 个步骤（前 3 步复用已有 Tool，后 3 步是编排器独有逻辑）。两个编排器职责不同：Agent 编排器负责推理循环，Workflow 编排器负责业务流程。
可能追问

"编排器和普通函数调用有什么区别？"

普通函数调用是同步的、固定的。编排器的每一步之间有状态传递、有阻断检查、有 Trace 记录。generate_learning_plan 在执行第 2 步后发现先修冲突——它不会继续执行第 3、4 步，而是立即返回 PREREQUISITE_CONFLICT 错误。编排器知道"什么时候该停"。

20架构 ToolAdapter 是做什么的？为什么不放在 ToolRegistry 里？▼
导师意图
考察对单一职责和关注点分离的理解。
推荐回答
ToolAdapter（to_openai_tools()）负责将 ToolRegistry 的元数据转换为 OpenAI Function Calling 的 Schema 格式——只有 20 行代码。不放在 ToolRegistry 里是因为：ToolRegistry 的职责是工具注册和调度，不关心下游格式。如果以后不用 OpenAI 格式而用 Anthropic 的 tool_use 格式，只需新增一个 Adapter 函数——ToolRegistry 完全不受影响。这就是单一职责。
21架构 你的 RuleBasedReActAgent 和 LLMReActAgent 是什么关系？▼
导师意图
考察是否设计了合理的继承/组合关系——是否可替换。
推荐回答
它们共享相同的 run(course_name, daily_hours, duration_days, user_knowledge) → ReActResult 接口。RuleBasedReActAgent 通过 EXECUTION_PLAN（预定义的 ToolStep 列表）确定执行序列，不需要 LLM。LLMReActAgent 通过 LLM Provider 动态推理决定下一步。AgentRunner 接受任意一个——注入哪个就用哪个，接口完全一致。这种设计让系统既可以离线运行（无需 API Key），也可以接 LLM 运行。
22架构 Prompt Loader 支持变量注入，为什么需要这个设计？▼
导师意图
考察对 Prompt 工程化的思考——是否让 Prompt 变成可配置的资源。
推荐回答
如果 Prompt 写死在代码里，修改 Prompt 就要改代码。Prompt Loader 支持 {{tool_list}}、{{workflow}}、{{current_date}}、{{user_input}}、{{output_schema}} 五个变量注入。工具列表从 ToolRegistry 实时获取——注册新 Tool 后 Prompt 自动更新。工作流描述从 Agent 的 EXECUTION_PLAN 生成——改工作流不需要改 Prompt 文件。这是 Prompt 的"配置化管理"。
23架构 你的 Scheduler 是怎么设计的？▼
导师意图
考察独立组件的设计质量——是否有清晰的接口和算法。
推荐回答
Scheduler 是独立的调度器类，接收模块列表和每日小时数，输出按天分配的 DailyPlan。核心算法是贪心打包：每天填充模块直到达到 daily_hours 上限，跨天模块自动拆分并标记 continuation。每 5 个学习日插入一个复习日，最后一个模块后插入评估日。Scheduler 不依赖 Course/Topic/Agent 等业务模型——它是纯调度组件，可以单独测试和替换。
24架构 ResourceProvider 为什么用 Provider 模式？▼
导师意图
考察对未来扩展的前瞻性设计——是否为后续优化预留了接口。
推荐回答
当前 MVP 用 HardCodedResourceProvider——基于关键词匹配的内置资源映射表。但接口定义是 ResourceProvider Protocol：get_resources(topic, difficulty) → list[Resource]。后续可以替换为 WebResourceProvider（调用在线课程 API）、RAGResourceProvider（向量检索知识库）、DBResourceProvider（从数据库查询）。编排器只依赖 Protocol，不依赖具体实现——替换 Provider 时编排器零改动。
25架构 你的数据模型层为什么用 Pydantic？▼
导师意图
考察技术选型的合理性。
推荐回答
三个原因。运行时验证：从 courses.json 加载数据时自动校验字段类型和约束——无效数据在入口就被拦截。零 alias：字段名从 JSON 到 Model 到 Tool 输出全链路一致，不需要 Field(alias=...)。JSON Schema 生成：model_json_schema() 可以自动生成 JSON Schema，后续可以用于自动生成 Tool 的 input_schema，消除手写重复。
3. ReAct 与 Tool Calling 10 个问题
26ReAct 你的 ReAct 机制具体如何实现？ReAct 是不是只是 Prompt？▼
导师意图
这是高频问题。考察是否真正实现了 ReAct 循环，而非仅在 Prompt 里写了"请思考"。
推荐回答
不只是 Prompt。ReAct 在我的系统中有两层实现。规则引擎模式：EXECUTION_PLAN 定义了 4 个 ToolStep，每个步骤包含 thought_template（推理模板）、params_builder（参数构建）、blocker_check（阻断检查）。Agent 遍历这些步骤，在每个步骤执行 THINKING → TOOL_EXECUTION → OBSERVATION 的状态转换，并产生 Trace 记录。LLM 模式：Agent 把 tools_schema 和对话历史发给 LLM，LLM 返回 tool_calls 或最终回答。Agent 执行 tool_calls 后将结果追加到对话历史，LLM 继续推理——直到 finish_reason="stop"。核心区别：我的 ReAct 是代码逻辑控制的循环，不是靠 Prompt 让 LLM "假装循环"。
可能追问

"规则模式的 ReAct 和 LLM 模式的 ReAct 有什么本质区别？"

规则模式是确定性序列——每次相同输入产生相同输出。LLM 模式是动态推理——LLM 根据上下文决定下一步。规则模式不需要 API Key，适合演示和测试；LLM 模式更灵活，适合处理非标准请求。

27ReAct Agent 如何决定调用哪个 Tool？这是谁完成的？▼
导师意图
考察 Tool 选择机制——区分规则模式和 LLM 模式的不同决策方式。
推荐回答
分两种模式。规则模式：由 EXECUTION_PLAN 预定义——ToolStep 列表规定了每一步调用哪个 Tool。这是确定性的，不涉及 LLM 推理。LLM 模式：由 DeepSeek LLM 动态决定——Agent 把 tools_schema 发给 LLM，LLM 根据对话上下文返回 tool_calls 列表（包含 tool_name 和 arguments）。Agent 收到 tool_calls 后通过 ToolRegistry.invoke() 执行。关键分离：LLM 决定"调用哪个"，Agent 负责"怎么调用"。
28ReAct Tool Calling 的完整流程是什么？▼
导师意图
考察是否能完整描述数据流——从用户输入到最终回答的每一步。
推荐回答
五个步骤。第一步：ToolAdapter 将 ToolRegistry 的元数据转换为 OpenAI Function Calling Schema。第二步：Agent 把 Schema + System Prompt + 用户消息发给 DeepSeek。第三步：DeepSeek 返回 tool_calls 列表（不是文本回答）。第四步：Agent 遍历 tool_calls，通过 registry.invoke(name, **args) 执行每个 Tool。第五步：执行结果以 role: "tool" 追加到对话历史，LLM 继续推理。循环直到 LLM 返回 finish_reason="stop" 的文本回答。
29ReAct LLM 负责什么，代码负责什么？▼
导师意图
考察职责分离是否清晰——不是 LLM 在控制一切。
推荐回答
LLM 负责语义理解——理解用户意图、决定调用哪个 Tool、传什么参数、如何解读 Tool 返回结果、何时给出最终回答。代码负责工程执行——ToolRegistry 调度工具、blocer_check 检查阻断条件、Trace 记录每一步、MAX_TOOL_ROUNDS 限制循环次数。LLM 提供灵活性，代码提供可靠性。即使 LLM 选错了工具，代码层的安全约束也不会让系统崩溃。
30ReAct 如果 LLM 选择了错误的工具怎么办？▼
导师意图
考察容错机制——系统是否能处理 LLM 的不当行为。
推荐回答
三层保护。第一，ToolRegistry.invoke() 内部有 try/except——工具执行异常被捕获为 INTERNAL_ERROR，以标准格式返回给 LLM，LLM 可以根据错误信息调整策略。第二，MAX_TOOL_ROUNDS=10——即使 LLM 反复选错工具，循环也不会无限进行。第三，blocker_check——如果工具返回了业务阻断（如先修不满足），Agent 立即终止，不会让 LLM 继续尝试。
31ReAct 你的 Agent 是否真正具备自主规划能力？▼
导师意图
考察是否清楚当前系统的能力边界——不夸大。
推荐回答
需要区分两个概念。规则模式下，Agent 的"规划"是预定义的——EXECUTION_PLAN 规定了 4 步序列，这不是自主规划，是确定性的工作流执行。LLM 模式下，Agent 可以在 tool_calls 层面做动态决策——LLM 根据 Tool 返回结果决定下一步，这有了一定的灵活性。但整体工作流（先查课程→检查先修→评估时间→生成计划）仍然是结构化的。不能说 Agent 具有"完全自主规划能力"——它是一个在结构化工作流中具有动态工具选择能力的 Agent。
32ReAct Thought → Action → Observation 在你的代码里对应什么？▼
导师意图
考察代码和理论的对应关系——不是只会在 PPT 上画图。
推荐回答
Thought 对应 step_def.format_thought(state)——从 ToolStep 的 thought_template 生成推理文本，并作为 TraceEntry 记录。Action 对应 self.registry.invoke(tool_name, **params)——通过 ToolRegistry 调度执行。Observation 对应 _summarize_output(result)——提取 Tool 返回结果的关键字段，检查 success 标志和 blocker_check。这三个阶段在 RuleBasedReActAgent.run() 的 for 循环中依次执行，每个阶段产生一条 TraceEntry。
33ReAct ReAct 和普通 Chain（顺序调用）有什么区别？▼
导师意图
考察对 ReAct 范式独特性（观察→反馈→调整）的理解。
推荐回答
普通 Chain 是 A→B→C→D 的线性执行，上一步的输出作为下一步的输入，但没有"根据结果调整行为"的能力。ReAct 的关键在于 Observation 阶段的反馈——Agent 检查 Tool 返回结果后可以决定：继续下一步（正常）、切换策略（LLM 模式下）、立即终止（阻断条件触发）。这个"根据观察结果改变执行路径"的能力是 ReAct 区别于普通 Chain 的核心。
34ReAct 你的系统支持并行 Tool 调用吗？▼
导师意图
考察对 Tool Calling 高级特性的认知——是否知道并行调用的场景和限制。
推荐回答
当前不支持并行 Tool 调用——EXECUTION_PLAN 是串行执行的，LLM 模式下也是逐个处理 tool_calls。但架构上可以扩展：OpenAI Function Calling 本身就支持在一次响应中返回多个 tool_calls，ToolRegistry 的 invoke() 是无状态的，理论上可以用 asyncio.gather() 并行执行。课程规划场景中——get_course_info 和 get_prerequisite 没有数据依赖，实际上可以并行。这是后续优化的一个方向。
35ReAct LLM 输出的 tool_calls 格式不稳定怎么办？▼
导师意图
考察对 LLM 输出不确定性的认知和应对。
推荐回答
这也是为什么系统同时保留了规则模式。规则模式完全不依赖 LLM 输出——EXECUTION_PLAN 是确定性的，适用于对可靠性要求高的场景。LLM 模式下，DeepSeek Provider 的 _parse_response() 做了容错——tool_calls 的 arguments 如果 JSON 解析失败，fallback 为空 dict。更完善的方案（如参数校验、重试机制）在后续计划中。
4. 工程实现 10 个问题
36工程 为什么设计 Trace？它是日志吗？▼
导师意图
高频问题。考察 Trace 的设计价值——是否理解可观测性对 Agent 系统的意义。
推荐回答
Trace 不是普通日志。日志记录"发生了什么"，Trace 记录"Agent 为什么这样做"。每条 TraceEntry 包含 8 个字段——不只是记录 Tool 调用，还记录了 Agent 在 THINKING 状态时的推理内容（thought）、选择的工具（selected_tool）、观察到的结果（tool_output）。成功路径 13 步，先修冲突 7 步，时间不足 10 步。Chatbot 的输出没有审计记录——错了你不知道为什么错。Agent 的 Trace 可以定位到每一步的决策依据。
可能追问

"Trace 会影响性能吗？"

每条 tool_output 做了摘要处理，只保留关键字段（success/data_keys/satisfied/feasible），不存储完整数据。整个 13 步 Trace 约几 KB，Python 内存中 dataclass，无磁盘 IO。性能影响可忽略。

37工程 如何处理 Tool 执行异常？▼
导师意图
考察异常处理体系的完整性。
推荐回答
三层异常处理。Tool 层：每个 Tool 内部 try/except，异常返回 {"success": false, "error": {"code": "INTERNAL_ERROR", ...}}——不会抛未捕获异常。Registry 层：invoke() 内部 try/except，Tool 函数如果抛出异常，Registry 捕获并包装为 INTERNAL_ERROR。Agent 层：检查 result["success"]——如果为 false，Agent 转入错误处理分支（记录 Trace、构建错误响应）。所有异常都有明确的 error_code，共 8 种类型。
38工程 如何避免 Agent 循环调用 Tool？▼
导师意图
考察对 Agent 安全性的工程思考。
推荐回答
两层防护。规则模式：EXECUTION_PLAN 只有 4 步——执行完就结束，不会循环。LLM 模式：MAX_TOOL_ROUNDS = 10——即使 LLM 反复请求 tool_calls，循环最多 10 轮就强制终止。这类似于操作系统的"时间片"机制——给 LLM 足够的灵活性，但不会给它无限的控制权。
39工程 如何测试 Agent 系统？▼
导师意图
考察对 Agent 系统测试策略的理解——Agent 的测试比普通函数复杂。
推荐回答
分三层测试。模型层（66 个测试）：测试 Pydantic 模型的验证、序列化、边界值。工具层（54 个测试）：单独测试每个 Tool 的输入输出、错误处理、边界情况——使用真实 courses.json 数据。Agent 集成层（42 个测试）：测试 AgentRunner 生命周期、成功路径/先修冲突/时间不足三种端到端场景、Mock Agent/Mock Registry 的依赖注入、Trace 记录的正确性。总计 162 个测试，覆盖率约 90%。
40工程 你的系统目前有哪些工程保证？▼
导师意图
考察对软件工程实践的全面理解。
推荐回答
五个方面。类型安全：所有公共接口有完整类型注解，Pydantic v2 运行时验证。依赖注入：AgentRunner 构造函数接受可选参数，测试时可注入 Mock。异常体系：8 个自定义异常类，每种有明确 error_code。测试覆盖：162 测试覆盖模型/工具/Agent 三层。可审计性：每步执行产生 TraceEntry，完整记录决策过程。另外代码有完整中文文档字符串，依赖方向严格向下，零循环导入。
41工程 DAG 感知递归具体怎么实现的？▼
导师意图
考察算法实现的细节——不是只画了概念图。
推荐回答
递归函数 _expand_prerequisites() 维护一个 visited: set[str]。进入节点时检查 name_lower in visited——如果 True，说明这个课程已被其他路径处理过（共享依赖），直接 return，不报错。如果用户已完成该课程（name in user_set），也直接 return——隐式满足其先修。最大递归深度 MAX_RECURSION_DEPTH=5 做兜底。这和检测循环依赖不同：循环依赖需要额外维护"当前路径栈"，而 visited 只做去重——因为先修图是有向无环图。
42工程 你的系统能否处理未知课程？▼
导师意图
考察错误处理——系统对异常输入的反应。
推荐回答
可以。如果用户输入的课程不在 courses.json 中，get_course_info 返回 {"success": false, "error": {"code": "COURSE_NOT_FOUND", "details": {"available_courses": [...]}}}。Agent 在 Observation 阶段检测到 success=false，立即终止并列出所有 10 门可用课程。规则模式和 LLM 模式都支持——规则模式通过 blocker_check 机制，LLM 模式通过检查 tool 返回结果的 success 字段。
43工程 如果 courses.json 损坏了怎么办？▼
导师意图
考察数据层的容错设计。
推荐回答
DataLoader 的 _read_json() 做了两层保护：文件不存在 → FileNotFoundError（含文件路径）；JSON 解析失败 → DataCorruptionError（含行号）。load_courses() 在 Pydantic 验证阶段也会捕获异常——每条数据验证失败时抛出 DataCorruptionError，details 中包含文件路径、索引和 course_id，精确定位问题记录。整个加载流程不会静默失败——任何问题都有明确的错误信息和定位信息。
44工程 你的测试中 Mock 是怎么用的？▼
导师意图
考察测试策略中的依赖隔离——是否真正做到了可测试的架构。
推荐回答
两种 Mock 场景。Mock Agent：创建一个返回预设 ReActResult 的 MockAgent，注入 AgentRunner——测试 Runner 的格式化和生命周期，不触发真实 Tool 调用。Mock Registry：创建一个 ToolRegistry 并注册返回固定数据的 Tool，注入 RuleBasedReActAgent——测试 Agent 的 ReAct 循环逻辑，不需要真实数据文件。这两类 Mock 测试验证了依赖注入的有效性——Runner 和 Agent 不依赖具体实现。
45工程 你的代码有 3707 行，对于一个 MVP 是不是太多了？▼
导师意图
考察代码行数的合理性——是否过度设计。
推荐回答
3707 行包含完整类型注解、中文文档字符串和 1815 行测试代码。去掉测试，核心源码约 1900 行，分布在 4 个层级 26 个文件中。对于一个包含 4 层架构、10 个 Pydantic 模型、6 个 Tool、ReAct 状态机、Tool Registry、LLMProvider 抽象的系统，这个规模是合理的。很多行是结构化的定义——比如 Pydantic 模型的 Field 声明、ToolEntry 的 Schema 定义——这些不是冗余，是类型安全保障。
5. 创新点和不足 5 个问题
46创新 你的项目创新点是什么？▼
导师意图
高频问题。考察是否能清晰区分"创新"和"功能"。
推荐回答
创新点是架构层面的，不是功能层面的。"能生成学习计划"不是创新——大模型也能做到。创新在于：为 LLM 构建了一个可靠的运行框架——ToolRegistry 实现 O(1) 动态工具调度、LLMProvider ABC 实现模型可替换、ReAct 状态机同时支持规则模式和 LLM 模式、blocker_check 实现业务安全约束、Trace 提供完整的执行审计。这些不是调 API 能得到的——是工程设计的价值。
可能追问

"这些设计和别人做的有什么不同？"

大多数 Agent 项目依赖 LangChain/AutoGPT 框架。我的系统从零实现——核心 ReAct 循环不到 200 行，你可以看到每一行在做什么。而且架构为扩展而设计——加 Tool、换 Provider 都只需注册或实现接口。这不是框架的"使用者"，是框架的"设计者"视角。

47创新 和 LangChain / Dify 相比有什么区别？为什么不用？▼
导师意图
高频问题。考察对业界方案的了解，以及自己实现的价值。
推荐回答
LangChain 是通用 Agent 框架——功能丰富但封装了很多细节。用 LangChain 只能证明"我会调 API"——Agent 的推理过程、Tool 调度、Trace 记录都在框架黑盒里。我自己实现的价值在于：每一行代码都可控、可解释、可调试。而且架构更轻量——核心 Agent 循环不到 200 行。Dify 是低代码平台——适合快速搭建应用，但架构灵活性不如自建。对于课程项目来说，理解原理比使用工具更重要。
可能追问

"如果以后需要更复杂的编排，你会考虑用 LangGraph 吗？"

会考虑。但接入 LangGraph 时不需要推倒重来——ToolRegistry 的注册和调度接口是稳定的，只需替换 Agent 层的编排逻辑。架构的分层设计就是为了这种局部替换。

48创新 当前系统最大的不足是什么？▼
导师意图
考察自我认知——是否清楚系统的边界和缺陷。
推荐回答
三个不足。第一，Tool 的 input_schema 目前是手写的 JSON Schema，和 Pydantic 模型有重复——应该从 model_json_schema() 自动生成。第二，课程数据只有 10 门——虽然对于 MVP 足够了，但覆盖面和真实课程规划场景有差距。第三，LLM 模式下没有参数校验——如果 LLM 传了错误的参数给 Tool，目前靠 Tool 自身验证，缺少前置的参数 Schema 校验。这些都是工程优化项，不影响核心架构。
49创新 如果继续开发，你会增加什么？▼
导师意图
考察对后续方向的规划——是否有清晰的演进路径。
推荐回答
三个方向按优先级排序。第一，Conversation Memory——支持多轮对话，让用户可以追问"跳过可选模块"或"改到 4 小时每天"。第二，RAG 资源检索——用向量数据库检索真实学习资源，替代当前的硬编码 ResourceProvider。第三，Multi-Agent 协作——让一个编排 Agent 把课程分析、时间规划、资源检索分发给专门的 Sub-Agent。当前 ToolRegistry 的设计已经为 Multi-Agent 留好了基础——每个 Sub-Agent 可以作为一个 Tool 注册。
50创新 如果重新设计这个项目，你会改变什么？▼
导师意图
考察反思能力——做完项目后能否识别设计中的不足。
推荐回答
三个改变。第一，Tool 的 Schema 从 Pydantic 模型自动生成而非手写——减少维护负担。第二，Scheduler 使用更智能的调度算法（如约束求解）替代简单贪心——当前算法会产生一些不够紧凑的安排。第三，更早引入依赖注入——项目早期 AgentRunner 是直接 new 组件的，后期重构为 DI 花了一些时间。但不改变的是：分层架构、Registry 模式、Provider 抽象——这三个设计决策在整个开发过程中没有返工，说明它们是正确的。
