常见导师提问 & 应答参考

Q1: 既然是规则化的，和传统 if/else 有什么本质区别？

应答：本质区别有四点：

  1. Tool Registry 解耦 — 传统方式用 if/elif 硬编码 Tool 调用。我们使用 Registry Pattern，新增 Tool 只需 registry.register(entry) 一行代码，无需修改主逻辑。
  2. 声明式执行计划 — EXECUTION_PLAN 是数据而非代码，blocker_check 是声明式的阻断条件，不是散落在各处的 if 判断。
  3. 完整 Trace — 每一步的 THINKING → TOOL → OBSERVATION → RESPONSE 都被记录，传统 if/else 做不到这种粒度的可观测性。
  4. LLM 就绪 — 只需覆写 _reason / _select_tool / _build_params / _check_blocker 4 个方法即可接入 LLM，架构设计遵循开闭原则。
Q2: 为什么不用 LangChain？

应答：LangChain 封装层次过多，debug 困难。我们的 Agent 核心代码仅 ~480 行（react_loop.py），完全自主实现 ReAct 范式。对于本科项目，这更能体现对核心原理的理解，而非调用第三方框架。同时，我们设计了与 LangChain 兼容的 Tool 接口（通过 tool_adapter.py），保留了集成能力。

Q3: LLM 模式下效果如何？

应答：LLMReActAgent 是 RuleBasedReActAgent 的子类，覆写了 4 个关键方法，通过 DeepSeek API 动态推理。启动方式：python demo.py --llm（需设置 DEEPSEEK_API_KEY）。离线模式保证答辩稳定性，LLM 模式展示扩展能力。二者共享同一套 Tool Registry 和接口协议。

Q4: 先修冲突检测如何处理循环依赖？

应答：使用 visited 集合在 DFS 递归过程中跟踪已访问节点。如果再次遇到已访问节点，直接跳过（共享依赖场景）而非报错。最大递归深度设为 5 层，超过则抛出 MaxRecursionDepthError。同时 courses.json 的先修图通过测试验证无循环（test_prerequisite_graph_consistency）。

Q5: 系统有什么局限性？

应答：三个主要局限：① 课程数据目前是静态 JSON（10门），未来可接入真实课程API；② 规则化Agent的推理路径固定，复杂场景需LLM模式；③ 缺少用户交互界面，目前仅命令行。这些都在未来工作计划中。
