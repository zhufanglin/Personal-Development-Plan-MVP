Background

为什么需要课程学习规划 Agent？

传统学习规划依赖人工经验，存在三大问题。

01

人工规划困难

学生需要自己搜索课程资料，向学长打听经验，难以确定合理学习顺序。不同人的建议差异大，缺少可靠的参考依据。

02

课程关系复杂

先修依赖形成有向无环图（DAG）——学 Spark 前需要 Python 和 Hadoop，学 Hadoop 又要 Linux 和 Java。人脑很难完整梳理多层依赖。

03

时间约束难评估

每天只能学 2 小时、只有 30 天——这个约束下某门课能不能学完？传统方式靠感觉判断，无法给出精确的时间可行性评估。

因此设计了一个 Agent：不是让 LLM 直接回答，而是让 LLM 调用工具查询真实数据

Architecture

系统架构：四层分离

Provider → Agent → Tools → Models，依赖方向严格向下，零循环导入。

1
Provider 层 — LLMProvider 抽象

封装大模型 API 调用。当前接 DeepSeek（OpenAI 兼容协议），预留 Claude/OpenAI 扩展点。Agent 代码零 SDK 依赖。

2
Agent 层 — ReAct 循环 + ToolRegistry

AgentRunner（总入口）、LLMReActAgent（ReAct 推理）、ToolRegistry（O(1) 工具调度）。RuleBasedReActAgent 支持离线运行。

3
Tools 层 — 6 个工具 + Scheduler + ResourceProvider

4 个课程规划 Tool + 2 个 Demo Tool。Scheduler 贪心调度每日计划，ResourceProvider 关键词匹配学习资源。

4
Models 层 — 10 个 Pydantic 模型

字段名与 courses.json 全链路一致，零 alias。提供运行时类型验证和序列化。

Tools

工具设计：4 个课程规划 Tool

Agent 通过 Tool Calling 获取信息，而非依赖模型记忆。每步都可追溯。

工具职责输入关键输出
get_course_info 查询课程元数据 course_name: str 学时 / 难度 / 模块列表 / 先修课
get_prerequisite 递归展开先修树 course_name + user_knowledge 缺失课程 / 总先修学时 (DAG 感知)
calculate_learning_time 评估时间可行性 course_name + daily_hours + duration_days 可行/不可行 + 3 种调整方案
generate_learning_plan 编排完整学习计划 以上全部参数 学习路径 + 每日计划 + 资源推荐
01

ToolRegistry — O(1) 调度

使用 Python dict 实现工具查找，零 if/elif 条件链。新增工具只需 registry.register(ToolEntry(...))，Agent 代码零改动。

02

LLMProvider — 模型可替换

Agent 只依赖 LLMProvider 抽象基类，零 SDK import。当前使用 DeepSeek（OpenAI SDK + base_url），换 Claude/OpenAI 只需实现新 Provider。

Reliability

异常处理与安全约束

六层安全约束保障 Agent 稳定运行，两个业务阻断点防止浪费计算资源。

输入验证

_validate_input() 在 Agent 启动前拦截非法参数。hours <0.5 / days <1 / 空课程名 → 立即拒绝。

DAG 感知递归

visited 集合处理共享依赖（如 Java 被多条路径引用），不误报循环错误。MAX_RECURSION_DEPTH=5。

LLM 循环上限

MAX_TOOL_ROUNDS=10 防止 Agent 陷入无限工具调用循环。Provider 层 5 种异常映射中文消息。

两阶段业务阻断

阻断 1: 先修冲突

get_prerequisite 返回 satisfied=false → 立即终止。输出缺失课程 + 预估先修学时。

示例: Spark → 缺 Python + Hadoop → Hadoop 缺 Linux + Java → 4门/112h → 终止

阻断 2: 时间不足

calculate_learning_time 返回 feasible=false → 立即终止。输出 3 种调整方案。

方案A: 延长周期 · 方案B: 增加每日时间 · 方案C: 跳过可选模块
